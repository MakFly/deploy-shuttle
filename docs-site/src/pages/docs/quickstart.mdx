This walkthrough covers two paths: deploying a new app and auditing an
existing server. Both assume you have `shuttle` installed locally. If not,
see [Install](/docs/install).

## Path A: From Code to Production

### 1. Init your project

```bash
cd my-laravel-app
shuttle init
```

`init` detects your stack (Laravel, Symfony, Next.js, Node API) and
generates a production-ready Dockerfile, docker-compose.yml, and
.dockerignore. It prompts for your app name, domain, and server IP.

### 2. Provision your VPS

```bash
shuttle provision
```

Connects to your server over SSH and bootstraps Docker Swarm, Caddy
(reverse proxy with auto-TLS), and UFW (firewall). One command, bare
Ubuntu to production-ready infrastructure.

### 3. Deploy

```bash
shuttle deploy
```

Builds the Docker image, pushes to your registry, and deploys with
zero-downtime rolling updates via Docker Swarm. Your app is live at
`https://your-domain.com` with TLS from Let's Encrypt.

### 4. Rollback (if needed)

```bash
shuttle rollback --yes
```

Reverts to the previous deployment. Works for all three strategies
(Swarm, Compose, Blue-Green).

## Path B: Audit an Existing Server

### 1. Scan your server

```bash
shuttle doctor --target root@your-server
```

The command opens an SSH session, runs the 43 checks, and prints a console
report grouped by severity. Sample output:

```text
Shuttle Doctor Report

Target: root@203.0.113.10
Score: 62/100 - Risky

Critical:
  [x] Public sensitive database listeners detected: 5432
High:
  [x] UFW is missing or inactive
  [x] No recent backup artifacts and no pg_dump cron entry detected
Medium:
  [x] sshd defaults to port 22
```

`doctor` exits with code `0` for a clean run, `1` for any critical
finding, or whatever you set with `--fail-below`. Use `--fail-below 80` in
CI to block merges when the score drops.

### 2. Preview a hardening plan

```bash
shuttle harden --target root@your-server --dry-run
```

Each finding becomes an action: a rationale, shell commands or notes, and
a flag that says whether it can be auto-applied safely. Auto-apply is
opt-in via `--apply --yes` and limited to safe, idempotent actions.

### 3. Generate a client-facing report

```bash
shuttle report --format html --output report.html
```

### 4. Wire it into CI

```yaml
- uses: MakFly/deploy-shuttle@v1
  with:
    target: ${{ secrets.SSH_TARGET }}
    ssh-private-key: ${{ secrets.SSH_PRIVATE_KEY }}
    fail-below: '80'
```

The Action exposes `score`, `level`, and `report` outputs and writes a
job summary. See [GitHub Action](/docs/github-action) for the full
input/output reference.

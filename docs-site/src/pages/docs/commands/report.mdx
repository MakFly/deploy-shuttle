`report` does not run any check; it formats an existing JSON report into
something you can paste into Notion, share via a link, or hand to a
client as a PDF.

## Usage

```bash
shuttle report --format <markdown|html|pdf> [flags]
```

## Flags

| Flag | Default | Description |
| --- | --- | --- |
| `--format markdown\|html\|pdf` | required | Output format. |
| `--input <path>` | `.shuttle/latest-report.json` | doctor JSON. |
| `--output <path>` | format-specific (`report.md`, `shuttle-report.html`, `report.pdf`) | Output file. |

## Examples

```bash
# Free: Markdown for handoff in PRs / Notion / chat.
shuttle report --format markdown --output report.md

# Pro: self-contained HTML with executive summary.
shuttle report --format html --output report.html

# Pro: PDF for the client handoff.
shuttle report --format pdf --output report.pdf
```

## What the report contains

Every format includes:

- Target, score, level, generated-at timestamp.
- Executive summary (one paragraph derived from severities + counts).
- Findings grouped by severity with remediation steps.
- Accepted risks (checks ignored via `.shuttle.yml`, with rationale).
- Compact evidence per finding (no raw stack traces in client view).
- Next-actions block (the recommended `harden`/`doctor` follow-ups).

The JSON evidence is **not** included in the client-facing body; you keep
that in `latest-report.json`.

## Tier

| Format | Tier |
| --- | --- |
| `markdown` | Free |
| `html` | Pro |
| `pdf` | Pro |

The Markdown report is enough to ship a project; HTML is for hosted
shareable links, PDF is for the client deliverable.

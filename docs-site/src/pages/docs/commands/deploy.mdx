`deploy` builds your Docker image, pushes it to the configured registry,
and deploys to your server using the strategy defined in `shuttle.yml`.
The default strategy is **Swarm** with zero-downtime rolling updates.

## Usage

```bash
shuttle deploy [flags]
```

## Flags

| Flag | Default | Description |
| --- | --- | --- |
| `--config <path>` | nearest `shuttle.yml` | Deploy configuration. |
| `--target <user@host[:port]>` | from config | Override the deploy target. |
| `--strategy <name>` | from config or `swarm` | One of `swarm`, `compose`, `blue-green`. |
| `--format console\|json` | `console` | Output format. |

## Strategies

### Swarm (default)

```yaml
deploy:
  strategy: swarm
```

Uses `docker stack deploy` with rolling updates. Services are updated
one at a time with a start-first order, meaning the new container starts
and passes its healthcheck before the old one is removed. If the new
container fails to start, Docker automatically rolls back.

Key behaviors:

- Zero-downtime via start-first rolling updates.
- Docker Swarm manages service state and desired replicas.
- Secrets are pushed as Docker Secrets (encrypted in Raft log, mounted
  as files in `/run/secrets/` inside the container, never on disk).
- Auto-rollback on healthcheck failure.
- Manual rollback with `shuttle rollback --yes`.

### Blue-Green

```yaml
deploy:
  strategy: blue-green
```

Runs two deployment slots (blue and green). Each deploy targets the
inactive slot. Once the new slot is healthy, Caddy's upstream is
switched via SIGUSR1 hot-reload -- no restart, no dropped connections.

Key behaviors:

- Two slots running in parallel during the switch.
- Caddy upstream switch is atomic (SIGUSR1).
- Instant rollback by swapping back to the previous slot.
- Ideal for apps that need pre-warming or long startup times.

### Compose

```yaml
deploy:
  strategy: compose
```

Uses `docker compose up -d` with the generated compose file. Simple and
suitable for staging or development environments.

Key behaviors:

- Standard docker compose lifecycle.
- State tracking for rollback support.
- Brief downtime during container replacement.

## Examples

```bash
# Deploy using the strategy from shuttle.yml.
shuttle deploy

# Override strategy for this deploy.
shuttle deploy --strategy compose

# Deploy to a specific target.
shuttle deploy --target deploy@staging.example.com
```

## WireGuard-only servers

When SSH is only reachable through a WireGuard network, set the server host
to the private VPN address and enable the VPN preflight in `shuttle.yml`:

```yaml
server:
  host: 10.8.0.12
  user: deploy
  port: 22
  vpn:
    required: true
    interface: wg0
    check_host: 10.8.0.12
    check_port: 22
```

`domain` should still be the public hostname used by Caddy/TLS. Only the
SSH host changes to the WireGuard address. When `vpn.required` is enabled,
Shuttle checks the WireGuard interface when `wg` is available, then verifies
that the configured host and port are reachable before opening SSH or the
reverse registry tunnel.

## What happens during a deploy

1. **Build** -- Docker image is built from the Dockerfile generated by
   `shuttle init` (or your custom Dockerfile).
2. **Push** -- Image is pushed to the configured registry.
3. **Deploy** -- The strategy-specific deploy command runs on the remote
   server.
4. **Health** -- The CLI waits for the healthcheck to pass.
5. **Done** -- The deploy state is recorded for rollback support.

## Pre-requisites

- Server provisioned with `shuttle provision` (or manually with Docker
  Swarm + Caddy).
- `shuttle.yml` configured with app name, domain, and server.
- Docker registry accessible from both your machine and the server.
